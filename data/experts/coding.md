# Coding & Code Review Expert Playbook

Starter playbook from common, widely-published software best-practices.

- Prefer clear, readable names over cleverness; code is read more than written.
- Keep functions small and single-purpose; one reason to change.
- Handle errors and edge cases explicitly; fail loudly, not silently.
- Write tests for behavior and contracts, not implementation details.
- Avoid premature optimization; measure before you tune.
- Prefer explicit interfaces over shared mutable state.
- Document the "why" in comments; let the code show the "what".
- Make changes small and reviewable; one concern per change.
